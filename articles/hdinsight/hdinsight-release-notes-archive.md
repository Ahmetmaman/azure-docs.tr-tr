# <a name="archived-release-notes"></a>Arşivlenmiş sürüm notları

> [!IMPORTANT]  
> Linux, HDInsight sürüm 3.4 ve üzerinde kullanılan tek işletim sistemidir. Daha fazla bilgi için [HDInsight sürüm makalesine](hdinsight-component-versioning.md)bakın.

## <a name="summary"></a>Özet

Azure HDInsight, Azure'da açık kaynak kodlu Apache Hadoop ve Apache Spark analitiği için kurumsal müşteriler arasında en popüler hizmetlerden biridir.

## <a name="release-date-12172019"></a>Yayın tarihi: 17.12.2019

Bu sürüm hem HDInsight 3.6 hem de 4.0 için geçerlidir.

### <a name="new-features"></a>Yeni özellikler

#### <a name="service-tags"></a>Hizmet etiketleri
Hizmet etiketleri, Azure hizmetlerine ağ erişimini kolayca kısıtlamanızı sağlayarak Azure sanal makinelerinin ve Azure sanal ağlarının güvenliğini basitleştirir. Ağ güvenlik grubu (NSG) kurallarınızda, belirli bir Azure hizmetine genel olarak veya Azure bölgesi başına trafik izni vermek veya reddetmek için hizmet etiketleri kullanabilirsiniz. Azure, her etiketin altında yatan IP adreslerinin bakımını sağlar. Ağ güvenliği grupları (NSG'ler) için HDInsight hizmet etiketleri, sağlık ve yönetim hizmetleri için IP adresleri gruplarıdır. Bu gruplar, güvenlik kuralı oluşturma karmaşıklığını en aza indirmeye yardımcı olur. HDInsight müşterileri Azure Portal, PowerShell ve REST API aracılığıyla hizmet etiketini etkinleştirebilir. Daha fazla bilgi için [Azure HDInsight için Ağ güvenlik grubu (NSG) hizmet etiketlerine](https://docs.microsoft.com/azure/hdinsight/hdinsight-service-tags)bakın.

#### <a name="custom-ambari-db"></a>Özel Ambari DB
HDInsight artık Apache Ambari için kendi SQL DB'nizi kullanmanıza olanak tanır. Bu özel Ambari DB'yi Azure portalından veya kaynak yöneticisi şablonu aracılığıyla yapılandırabilirsiniz.  Bu özellik, işleme ve kapasite gereksinimleriniz için doğru SQL DB'yi seçmenize olanak tanır. Ayrıca, iş büyüme gereksinimlerini karşılamak için kolayca yükseltebilirsiniz. Daha fazla bilgi için bkz: [ÖZEL Ambari DB ile HDInsight kümeleri ayarla.](hdinsight-custom-ambari-db.md)

![Özel Ambari DB](./media/hdinsight-release-notes/custom-ambari-db.png)

#### <a name="transport-layer-security-tls-12-enforcement"></a>Taşıma Katmanı Güvenliği (TLS) 1.2 uygulama
Aktarım Katmanı Güvenliği (TLS) ve Güvenli Soket katmanı (SSL), bir bilgisayar ağı üzerinden iletişim güvenliği sağlayan şifreleme protokolleridir. Daha fazla bilgi için [Taşıma Katmanı Güvenliği'ne](https://en.wikipedia.org/wiki/Transport_Layer_Security#SSL_1.0.2C_2.0_and_3.0)bakın. Azure HDInsight kümeleri ortak HTTPS uç noktalarında TLS 1.2 bağlantılarını kabul etse de, TLS 1.1 hala eski istemcilerle geriye dönük uyumluluk için desteklenir.

Bir sonraki sürümden itibaren, yeni HDInsight kümelerinizi yalnızca TLS 1.2 bağlantılarını kabul edecek şekilde kabul edebilecek ve yapılandırabilirsiniz. 

30/6/2020 tarihinden itibaren yılın ilerleyen saatlerinde Azure HDInsight, tüm HTTPS bağlantıları için TLS 1,2 veya sonraki sürümleri uygular. Tüm müşterilerinizin TLS 1.2 veya sonraki sürümlerini işlemeye hazır olduğundan emin olmamızı öneririz.

#### <a name="moving-to-azure-virtual-machine-scale-sets"></a>Azure sanal makine ölçek kümelerine geçme
HDInsight artık kümeyi sağlamak için Azure sanal makinelerini kullanıyor. Şubat 2020'den itibaren (kesin tarih daha sonra iletilecek), HDInsight bunun yerine Azure sanal makine ölçek kümelerini kullanır. [Azure sanal makine ölçek kümeleri](https://docs.microsoft.com/azure/virtual-machine-scale-sets/overview)hakkında daha fazla bilgi edinin.

#### <a name="esp-spark-cluster-node-size-change"></a>ESP Kıvılcım küme düğümü boyutu değişikliği 
Yaklaşan sürümde:
- ESP Spark kümesi için izin verilen minimum düğüm boyutu Standard_D13_V2 olarak değiştirilir. 
- A serisi VM'ler, göreceli olarak düşük CPU ve bellek kapasitesi nedeniyle ESP küme sorunlarına neden olabileceğinden, yeni ESP kümeleri oluşturmak için amortismana esas alınacaktır.

#### <a name="hbase-20-to-21"></a>HBase 2.0 ile 2.1 arasında
Yaklaşan HDInsight 4.0 sürümünde, HBase sürümü sürüm 2.0'dan 2.1'e yükseltilecektir.

### <a name="new-features"></a>Yeni özellikler

#### <a name="hdinsight-identity-broker-hib-preview"></a>HDInsight Kimlik Aracısı (HIB) (Önizleme)

HDInsight Identity Broker (HIB), kullanıcıların çok faktörlü kimlik doğrulama (MFA) kullanarak Apache Ambari'de oturum açmalarını ve Azure Active Directory Domain Services 'de (AAD-DS) parola kalıplarına gerek kalmadan gerekli Kerberos biletlerini almalarını sağlar. Şu anda HIB yalnızca Azure Kaynak Yönetimi (ARM) şablonu aracılığıyla dağıtılan kümeler için kullanılabilir.

#### <a name="kafka-rest-api-proxy-preview"></a>Kafka Rest API Proxy (Önizleme)

Kafka Rest API Proxy, güvenli AAD yetkilendirmesi ve OAuth protokolü ile Kafka kümesi ile yüksek kullanılabilir REST proxy'nin tek tıklamayla dağıtılmasını sağlar. 

#### <a name="auto-scale"></a>Otomatik ölçeklendirme

Azure HDInsight için otomatik ölçeklendirme artık apache Spark ve Hadoop küme türleri için genel olarak tüm bölgelerde kullanılabilir. Bu özellik, büyük veri analitiği iş yüklerinin daha uygun maliyetli ve üretken bir şekilde yönetilmesini mümkün kılar. Artık HDInsight kümelerinizin kullanımını optimize edebilir ve yalnızca ihtiyacınız olan parayı ödeyebilirsiniz.

Gereksinimlerinize bağlı olarak, yük tabanlı ve zamanlama tabanlı otomatik ölçekleme arasında seçim yapabilirsiniz. Yük tabanlı Otomatik Ölçek, geçerli kaynak gereksinimlerine göre küme boyutunu yukarı ve aşağı ölçeklendirebilirken, zamanlama tabanlı Otomatik Ölçek önceden tanımlanmış bir zamanlamaya göre küme boyutunu değiştirebilir. 

HBase ve LLAP iş yükü için otomatik ölçeklendirme desteği de genel önizlemedir. Daha fazla bilgi için bkz: [Azure HDInsight kümelerini otomatik olarak ölçeklendirin.](https://docs.microsoft.com/azure/hdinsight/hdinsight-autoscale-clusters)

#### <a name="hdinsight-accelerated-writes-for-apache-hbase"></a>HDInsight Hızlandırılmış Apache HBase için yazıyor 

Hızlandırılmış Yazmalar özelliği Apache HBase Önceden Yazılan Günlük (WAL) performansını geliştirmek için Azure premium SSD yönetilen disklerini kullanır. Daha fazla bilgi için bkz. [Apache HBase için Azure HDInsight Hızlandırılmış Yazmalar](https://docs.microsoft.com/azure/hdinsight/hbase/apache-hbase-accelerated-writes).

Artık müşteriler Ambari için doğru SQL DB'yi seçebilir ve kendi iş büyüme gereksinimlerine göre kolayca yükseltebilirler. Dağıtım, Azure Kaynak Yöneticisi şablonuyla yapılır. Daha fazla bilgi için bkz: [ÖZEL Ambari DB ile HDInsight kümeleri ayarla.](https://docs.microsoft.com/azure/hdinsight/hdinsight-custom-ambari-db)

F serisi sanal makineler(VM'ler) Hafif işleme gereksinimleri ile HDInsight ile başlamak için iyi bir seçimdir. Daha düşük bir saatlik liste fiyatına sahip olan F Serisi, vCPU başına Azure İşlem Birimi (ACU) açısından fiyat-performans alanında Azure portföyündeki en iyi seçenektir. ### <a name="known-issues"></a>Bilinen sorunlar

#### <a name="replication-is-broken-for-secure-hbase-with-default-installation"></a>Varsayılan yükleme ile Güvenli HBase için çoğaltma bozuldu

HDInsight 4.0 için aşağıdaki adımları yapın:

1. Kümeler arası iletişimi etkinleştirin.
1. Etkin başlık düğümünde oturum açın.
1. Aşağıdaki komutla çoğaltmayı etkinleştirmek için bir komut dosyası indirin:

    ```
    sudo wget https://raw.githubusercontent.com/Azure/hbase-utils/master/replication/hdi_enable_replication.sh
    ```
1. Şu komutu yazın: `sudo kinit <domainuser>`.
1. Komut dosyasını çalıştırmak için aşağıdaki komutu yazın:

    ```
    sudo bash hdi_enable_replication.sh -m <hn0> -s <srclusterdns> -d <dstclusterdns> -sp <srcclusterpasswd> -dp <dstclusterpasswd> -copydata
    ```
HDInsight 3.6 için aşağıdakileri yapın:

1. Etkin HMaster ZK'da oturum açın.
1. Aşağıdaki komutla çoğaltmayı etkinleştirmek için bir komut dosyası indirin:
    ```
    sudo wget https://raw.githubusercontent.com/Azure/hbase-utils/master/replication/hdi_enable_replication.sh
    ```
1. Şu komutu yazın: `sudo kinit -k -t /etc/security/keytabs/hbase.service.keytab hbase/<FQDN>@<DOMAIN>`.
1. Aşağıdaki komutu yazın:

    ```bash
    sudo bash hdi_enable_replication.sh -s <srclusterdns> -d <dstclusterdns> -sp <srcclusterpasswd> -dp <dstclusterpasswd> -copydata
    ```

#### <a name="phoenix-sqlline-stops-working-after-migrating-hbase-cluster-to-hdinsight-40"></a>Phoenix Sqlline HDInsight 4.0 HBase küme sini geçtikten sonra çalışmayı durdurur

Aşağıdaki adımları yapın:

1. Aşağıdaki Phoenix tablolarını bırakın:
    1. `SYSTEM.FUNCTION`
    1. `SYSTEM.SEQUENCE`
    1. `SYSTEM.STATS`
    1. `SYSTEM.MUTEX`
    1. `SYSTEM.CATALOG`
1. Tablolardan herhangi birini silerseniz, tablolara bağlantı temizlemek için HBase'i yeniden başlatın.
1. `sqlline.py` komutunu yeniden çalıştırın. Phoenix, adım 1'de silinen tüm tabloları yeniden oluşturur.
1. HBase verileriniz için Phoenix tablolarını ve görünümlerini yeniden oluşturun.

#### <a name="phoenix-sqlline-stops-working-after-replicating-hbase-phoenix-metadata-from-hdinsight-36-to-40"></a>Phoenix Sqlline HDInsight 3.6 ile 4.0 HBase Phoenix meta verilerini çoğalttınktan sonra çalışmayı durdurur

Aşağıdaki adımları yapın:

1. Çoğaltma yapmadan önce, hedef 4.0 kümesine `sqlline.py`gidin ve çalıştırın. Bu komut, phoenix `SYSTEM.MUTEX` tabloları `SYSTEM.LOG` gibi oluşturur ve sadece 4.0 var.
1. Aşağıdaki tabloları bırakın:
    1. `SYSTEM.FUNCTION`
    1. `SYSTEM.SEQUENCE`
    1. `SYSTEM.STATS`
    1. `SYSTEM.CATALOG`
1. HBase çoğaltmasını başlatma

#### <a name="mahout"></a>Mahout

HDP-2.3.x ve 2.4.x'te Mahout'un belirli bir Apaçi salınımını sevkiyat yerine, Apache Mahout gövdesinde belirli bir revizyon noktasına senkronize ettik. Bu revizyon noktası 0.9.0 sürümünden sonra, ancak 0.10.0 sürümünden öncedir. Bu, 0.9.0 sürümü nde çok sayıda hata düzeltmesi ve işlevsel geliştirme sağlar, ancak 0.10.0'da yeni Spark tabanlı Mahout'a tam dönüşümden önce Mahout işlevinin kararlı bir şekilde serbest bırakılmasını sağlar.

HDP 2.3.x ve 2.4.x'te Mahout için seçilen revizyon noktası, 19 Aralık 2014 itibarıyla Apache Mahout'un "mahout-0.10.x" şubesinden, GitHub'da 0f037cb03e7c096 revizyonundan dır.

HDP-2.5.x ve 2.6.x'te , "commons-httpclient" kitaplığını Mahout'tan kaldırdık çünkü olası güvenlik sorunları olan eski bir kütüphane olarak gördük ve Mahout'taki Hadoop-Client'ı HDP-2.5'te kullanılan 2.7.3 sürümüne yükselttik. Sonuç olarak:

-   Daha önce derlenmiş Mahout işlerinin HDP-2.5 veya 2.6 ortamında yeniden derlenmesi gerekecektir.

-   Bazı Mahout işlerinin "classNotFoundException" veya "load class" hatalarıyla "org.apache.commons.httpclient", "net.java.dev.jets3t" veya ilgili sınıf adı önekleri ile ilgili hatalarla karşılaşabilme olasılığı küçüktür. Bu hatalar olursa, eski kitaplıktaki güvenlik sorunları riski ortamınızda kabul edilebilirse, iş için gerekli kavanozları sınıf yolunuza el ile yükleyip yüklememenizi düşünebilirsiniz.

-   Bazı Mahout işlerinin, ikili uyumluluk sorunları nedeniyle, Mahout'un hbase istemci kodunda hadoop-ortak kitaplıklara yapılan aramalarda çökmelerle karşılaşabileceği daha da küçük bir olasılık vardır. Ne yazık ki, güvenlik sorunları olabilir Mahout HDP-2.4.2 sürümü, geri dönmek dışında bu sorunu çözmek için hiçbir yolu yoktur. Yine, bu çok sıradışı olmalıdır ve herhangi bir Mahout iş paketi nde gerçekleşmesi olası değildir.

- **RangerUI: İlke formunda girilen politika koşulu metninin kaçışı**
    
  **Etkilenen Bileşen:** Ranger
    
  **Sorunun Açıklaması**
    
  Bir kullanıcı özel ilke koşulları yla ilke oluşturmak isterse ve ifade veya metin özel karakterler içeriyorsa, ilke zorlama çalışmaz. Özel karakterler, ilkeyi veritabanına kaydetmeden önce ASCII'ye dönüştürülür.
    
  **Özel Karakterler:**  &  &lt; &gt; \` " '
    
  Örneğin, koşul tags.öznitelikleri\['type'\]='abc' ilke kaydedildikten sonra aşağıdaki dönüştürülür.
    
  etiketler.attds\[&\#x27;dsds \#&x27; \] \#x27;cssdfs&x27; =& \#
    
  İlkeyi düzenleme modunda açarak bu karakterlerle ilke koşulunu görebilirsiniz.
    
  **Geçi -ci çözüm**
    
  - **Seçenek \#1: Ranger Rest API ile politika oluşturma/güncelleme**
        
      REST URL:&lt;&gt;http:// ana bilgisayar :6080/service/plugins/policies
        
      **İlke koşulu yla ilke oluşturma:**
        
      Aşağıdaki örnekte \`etiketleri etiketleri ile\` politika oluşturma \`ve\` politika koşulu astags.attr\['type'\]=='abc' ile genel gruba select, update, create, drop, alter, index, lock, all gibi tüm kovan bileşeni izinlerini seçerek atayacaktır.
        
      **Örnek:**
        
      ```bash
        curl -H "Content-Type: application/json" -X POST http://localhost:6080/service/plugins/policies -u admin:admin -d '{"policyType":"0","name":"P100","isEnabled":true,"isAuditEnabled":true,"description":"","resources":{"tag":{"values":["tags-test"],"isRecursive":"","isExcludes":false}},"policyItems":[{"groups":["public"],"conditions":[{"type":"accessed-after-expiry","values":[]},{"type":"tag-expression","values":["tags.attr['type']=='abc'"]}],"accesses":[{"type":"hive:select","isAllowed":true},{"type":"hive:update","isAllowed":true},{"type":"hive:create","isAllowed":true},{"type":"hive:drop","isAllowed":true},{"type":"hive:alter","isAllowed":true},{"type":"hive:index","isAllowed":true},{"type":"hive:lock","isAllowed":true},{"type":"hive:all","isAllowed":true}]}],"denyPolicyItems":[],"allowExceptions":[],"denyExceptions":[],"service":"tagdev"}'
      ```
        
      **Varolan ilkeyi ilke koşuluyla güncelleştirin:**
        
      Aşağıdaki örnek, politikayı \`etiketler-test\` olarak etiketlerle \`günceller ve ilke koşulu astags.attr\` \['type'\]=='abc' ile seç, güncelleştir, oluştur, bırak, değiştir, dizini, kilidi, tümünü seçerek genel gruba atar.
        
      REST URL:&lt;http://&gt;ana bilgisayar adı :6080/service/plugins/policies/&lt;policy-id&gt;
        
      **Örnek:**

Daha yeni bir HDInsight sürümüne yükseltme ile ilgili en iyi uygulamalar için [lütfen bu](https://docs.microsoft.com/azure/hdinsight/hdinsight-upgrade-cluster) kılavuzu izleyin.
